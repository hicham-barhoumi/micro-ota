# micro-ota

Simple, fast, reliable Over-The-Air updates for MicroPython. Push code to your ESP32 over WiFi without touching a USB cable.

---

## How it works

A small OTA server runs on the device in a background thread. The host CLI connects to it over TCP/WiFi, sends only the files that changed (diff based on SHA-256), and the device commits them atomically then resets.

```
[ PC ]  python3 host/uota.py fast
           │
           │  TCP :2018
           ▼
[ ESP32 ]  ota.py  ←  boot_guard.py
           │
           ▼
        main.py  (your app)
```

```

├── device/                   # Code that runs on the ESP32
│   ├── ota.py                # OTA server (protocol, staging, commit)
│   ├── boot_guard.py         # Crash counter, rollback hook
│   └── transports/
│       └── wifi_tcp.py       # WiFi connect + TCP server
│
├── host/                     # Code that runs on the PC
│   ├── uota.py               # CLI entry point
│   ├── manifest.py           # SHA-256 manifest builder
│   ├── bootstrap.py          # First-time serial uploader
│   └── transports/
│       ├── wifi_tcp.py       # TCP client
│       └── serial.py         # Raw REPL protocol (bootstrap only)
│
├── main.py                   # Your application (uploaded via OTA)
├── ota.json                  # Configuration
└── requirements.txt          # Host dependencies
```

## Quick start

### 1. Install host dependencies

```bash
pip install pyserial esptool
```

### 2. Configure `ota.json`

```json
{
    "version": "1.0.0",
    "hostname": "192.168.1.100",
    "port": 2018,
    "ssid": "MyWiFi",
    "password": "MyPassword",
    "serialPort": "/dev/ttyUSB0"
}
```

> Use the device IP address for `hostname` — mDNS (`micropython.local`) requires network support that may not be available.

### 3. Bootstrap the device (first time, via USB)

Connect the ESP32 via USB, then:

```bash
python3 host/uota.py bootstrap
```

This uploads the OTA library to the device over the raw REPL serial protocol. You only need to do this once. After this, all updates happen over WiFi.

### 4. Push your app

```bash
# Push only fastOtaFiles (fast, for iterating on main.py)
python3 host/uota.py fast

# Push all managed files
python3 host/uota.py full

# Wipe device first, then push everything
python3 host/uota.py full --wipe
```

## CLI reference

```
python3 host/uota.py <command> [options]
```

| Command | Description |
|---|---|
| `bootstrap` | First-time upload of OTA library via serial (USB required) |
| `fast` | Push `fastOtaFiles` to the device over WiFi |
| `full` | Push all managed files to the device over WiFi |
| `version` | Read the current version string from the device |
| `terminal` | Interactive command shell over WiFi |
| `flash` | Flash MicroPython firmware via esptool *(Phase 3, stub only)* |

### Global options

All WiFi commands accept:

| Option | Default | Description |
|---|---|---|
| `--host HOST` | `ota.json hostname` | Device IP or hostname |
| `--port PORT` | `ota.json port` | TCP port |
| `--version VER` | `ota.json version` | Version string to embed |

Bootstrap options:

| Option | Default | Description |
|---|---|---|
| `--port PORT` | `ota.json serialPort` | Serial port |
| `--baud BAUD` | `115200` | Baud rate |

`full` also accepts `--wipe` to delete all user files on the device before uploading.

## `ota.json` reference

```json
{
    "version": "1.0.0",
    "hostname": "192.168.1.100",
    "port": 2018,
    "ssid": "MyWiFi",
    "password": "MyPassword",
    "serialPort": "/dev/ttyUSB0",
    "transports": ["wifi_tcp"],
    "managed_paths": ["*.py", "lib/**", "*.json"],
    "excludedFiles": [
        ".git/**",
        "boot.py",
        "ota.json",
        "host/**",
        "device/**"
    ],
    "fastOtaFiles": ["main.py"],
    "fullOtaFiles": ["lib/**", "*.py", "*.json"]
}
```

| Key | Description |
|---|---|
| `version` | Version string written to the device after each OTA |
| `hostname` | Device IP or hostname for WiFi connection |
| `port` | TCP port the OTA server listens on (default: `2018`) |
| `ssid` / `password` | WiFi credentials stored on the device |
| `serialPort` | Serial port for bootstrap (auto-detected if empty) |
| `transports` | Ordered list of transports to try (`wifi_tcp` supported) |
| `managed_paths` | Glob patterns for files that can be managed by OTA |
| `excludedFiles` | Glob patterns to always exclude from OTA uploads |
| `fastOtaFiles` | Files uploaded by `uota fast` |
| `fullOtaFiles` | Additional files uploaded by `uota full` |

> `boot.py` must stay in `excludedFiles`. It is generated by `bootstrap` and starts the OTA server — overwriting it via OTA will break the next boot.

## Terminal mode

`uota terminal` opens an interactive shell directly to the device over WiFi. Useful for inspecting the filesystem without connecting USB.

```
$ python3 host/uota.py terminal
Terminal mode. Type 'exit' to quit.
Commands: ping, version, ls [path], get <path>, rm <path>, reset, wipe

$ version

$ ls /

transports

ota.json
$ ls /lib

ok
$ exit
```

## OTA protocol

The device accepts one command per TCP connection on port 2018.

| Command | Response | Description |
|---|---|---|
| `ping` | `pong` | Liveness check |
| `version` | `{"version":"x.y.z"}` | Read installed version |
| `ls [path]` | filenames, one per line | List directory |
| `get <path>` | `<size>\n<binary>` | Download a file |
| `rm <path>` | `ok` / `error: ...` | Delete a file |
| `reset` | `ok` then resets | Soft reset the device |
| `wipe` | `ok` | Delete all user files, keep OTA library |
| `start_ota` | `ready` | Begin an OTA session (see below) |

### OTA session (after `start_ota → ready`)

```
host → device
──────────────────────────────────────────────
manifest <size>\n<json>              → ok
file <name>;<size>;<sha256>\n<bin>   → ok / sha256_mismatch
file ...                             → ok
end_ota                              → ok  (atomic commit + reset)

  -- or at any point --
abort                                → aborted  (staging discarded)
```

Files are staged in `/ota_stage/` during the session. On `end_ota`:
1. Old manifest is diffed against the new one — files removed from the manifest are deleted from the device.
2. Staged files are moved atomically to their final paths.
3. New manifest and version are written.
4. Device resets.

If the connection drops mid-session, the staging directory is cleaned up on the next boot.

## Boot guard

`boot_guard.py` tracks consecutive unclean boots. On every boot it increments a crash counter stored in `/ota_boot_state.json`. The OTA server calls `mark_clean()` once it is running, which resets the counter.

If the device crashes 3 times before `mark_clean()` is called, a warning is printed on the serial console. In Phase 3 this will trigger an automatic firmware rollback via `esp32.Partition`.

To recover a crash-looping device, re-run bootstrap via USB:

```bash
python3 host/uota.py bootstrap
```

## Writing your app

`main.py` is your application entry point. The OTA server runs in a background thread started by `boot.py`. Your app runs in the main thread.

If your app is CPU-intensive (tight loops, no blocking I/O), add a short startup delay so the OTA thread has time to connect to WiFi before your loop takes over:

```python
# main.py
import time
time.sleep(2)   # let OTA server connect to WiFi

# your app here
```

Alternatively, connect to WiFi yourself before starting the heavy work and the OTA thread will reuse the existing connection automatically.
