# micro-ota

Simple, fast, reliable Over-The-Air updates for MicroPython. Push code to your ESP32 over WiFi or USB serial, and talk to it in real time over a persistent side channel.

---

## How it works

Three services run on the device in background threads at boot:

```
[ PC ]  python3 host/uota.py fast          python3 host/remoteio.py listen
           │  TCP :2018  (OTA protocol)          │  TCP :2019  (RemoteIO)
           ▼                                     ▼
[ ESP32 ]  ota.py  ──────────────────────  remoteio.py
           │             boot_guard.py
           ▼
        main.py  (your app)
```

- **OTA server** (port 2018): accepts file pushes, applies them atomically, resets.
- **RemoteIO server** (port 2019): forwards `print()` output to the host and handles RPC calls registered with `@remoteio.on('name')`.
- **Boot guard**: tracks crash counts so a bad update cannot brick the device.

Both OTA and RemoteIO can also run over **USB serial** via raw REPL injection — no WiFi needed for development.

## Quick start

### 1. Install host dependencies

```bash
pip install pyserial esptool
```

### 2. Configure `ota.json`

```json
{
    "version": "1.0.0",
    "hostname": "192.168.1.100",
    "port": 2018,
    "remoteioPort": 2019,
    "ssid": "MyWiFi",
    "password": "MyPassword",
    "serialPort": "/dev/ttyUSB0"
}
```

> Use the device IP address for `hostname` — mDNS names require network support that may not be available on all setups.

### 3. Bootstrap the device (first time, via USB)

Connect the ESP32 via USB, then:

```bash
python3 host/uota.py bootstrap
```

This uploads the OTA library, RemoteIO server, and a generated `boot.py` over the raw REPL serial protocol. You only need to do this once.

### 4. Push your app

```bash
# Push only fastOtaFiles (fast iteration on main.py)
python3 host/uota.py fast

# Push all managed files
python3 host/uota.py full

# Wipe device first, then push everything
python3 host/uota.py full --wipe

# Use USB serial instead of WiFi
python3 host/uota.py fast --transport serial
```

## OTA CLI reference

```
python3 host/uota.py <command> [options]
```

| Command | Description |
|---|---|
| `bootstrap` | First-time upload of OTA + RemoteIO library via serial (USB required) |
| `fast` | Push `fastOtaFiles` (default: `main.py`) |
| `full` | Push all managed files |
| `version` | Read the current version string from the device |
| `terminal` | Interactive OTA command shell |
| `flash` | Flash MicroPython firmware via esptool |

### Transport options

All `fast`, `full`, `version`, and `terminal` commands accept:

| Option | Default | Description |
|---|---|---|
| `--host HOST` | `ota.json hostname` | Device IP or hostname |
| `--port PORT` | `ota.json port` | TCP port |
| `--transport wifi\|serial` | `wifi` | Transport to use |
| `--version VER` | `ota.json version` | Version string to embed |

`bootstrap` accepts `--serial-port PORT` and `--baud BAUD`.
`full` also accepts `--wipe` to delete all user files before uploading.

## RemoteIO

RemoteIO is a persistent side-channel that runs on port 2019 alongside the OTA server. It does two things:

1. **Print forwarding** — all `print()` calls from the device are forwarded to the connected host in real time via `os.dupterm()`. No changes to your app code needed.
2. **RPC** — the host can call named handlers registered on the device with `@remoteio.on('name')`.

### CLI

```bash
# Stream all device print() output to the terminal
python3 host/remoteio.py listen

# Call a built-in handler
python3 host/remoteio.py call ping
python3 host/remoteio.py call free_mem
python3 host/remoteio.py call version

# Call a custom handler with arguments
python3 host/remoteio.py call echo msg=hello
```

### Python API

```python
from host.remoteio import RemoteIOClient

with RemoteIOClient('192.168.1.100') as rio:
    print(rio.call('ping'))          # 'pong'
    print(rio.call('free_mem'))      # 98304  (bytes)
    print(rio.call('uptime_ms'))     # 12345  (ms since boot)
    print(rio.call('version'))       # {'version': '1.0.0'}

# Stream prints while also making calls
import threading
with RemoteIOClient('192.168.1.100') as rio:
    threading.Thread(target=rio.listen, daemon=True).start()
    result = rio.call('sensor_data')
```

### Registering handlers on the device

In your `main.py` or any module loaded at boot:

```python
import remoteio

@remoteio.on('sensor_data')
def _():
    return {'temp': read_temp(), 'humidity': read_hum()}

@remoteio.on('set_led')
def _(state=False):
    led.value(state)
    return 'ok'

@remoteio.on('echo')
def _(msg=''):
    print('echo called with:', msg)  # this print goes to the host too
    return msg
```

### Built-in handlers

| Name | Arguments | Returns |
|---|---|---|
| `ping` | — | `'pong'` |
| `version` | — | `{"version": "x.y.z"}` |
| `free_mem` | — | free heap bytes (int) |
| `uptime_ms` | — | ms since boot (int) |

### Protocol

Newline-delimited JSON over TCP port 2019 (persistent connection):

```
Device → Host:  {"t": "print",  "d": "text"}
Host   → Device:{"t": "call",   "id": N, "name": "cmd", "args": {...}}
Device → Host:  {"t": "resp",   "id": N, "r": result}
Device → Host:  {"t": "err",    "id": N, "e": "error message"}
```

## OTA protocol

The OTA server (port 2018) handles one command per TCP connection on WiFi, or a persistent connection on serial.

| Command | Response | Description |
|---|---|---|
| `ping` | `pong` | Liveness check |
| `version` | `{"version":"x.y.z"}` | Read installed version |
| `ls [path]` | filenames, one per line | List directory |
| `get <path>` | `<size>\n<binary>` | Download a file |
| `rm <path>` | `ok` / `error: ...` | Delete a file |
| `reset` | `ok` then resets | Soft reset the device |
| `wipe` | `ok` | Delete all user files, keep OTA library |
| `start_ota` | `ready` | Begin an OTA session (see below) |

### OTA session

```
host → device
──────────────────────────────────────────────
manifest <size>\n<json>              → ok
file <name>;<size>;<sha256>\n<bin>   → ok / sha256_mismatch
...
end_ota                              → ok  (atomic commit + reset)

  -- or at any point --
abort                                → aborted  (staging discarded)
```

Files are staged in `/ota_stage/` during the session. On `end_ota`:
1. Old manifest is diffed against the new one — files no longer in the manifest are deleted from the device.
2. Staged files are moved atomically to their final paths.
3. New manifest and version are written.
4. Device resets.

If the connection drops mid-session, staging is cleaned up on the next boot.

## Serial OTA transport

Both `uota fast/full` and the OTA terminal work over USB serial — no WiFi needed:

```bash
python3 host/uota.py fast --transport serial
python3 host/uota.py terminal --transport serial
```

The host enters the raw REPL, injects a minimal OTA server inline, and then speaks the standard micro-ota protocol over the UART. The inline server uses `sys.stdout.buffer` / `sys.stdin.buffer` for binary-clean transfer.

Serial port is auto-detected from connected ESP32 devices; override with `--serial-port /dev/ttyUSB0`.

## `ota.json` reference

```json
{
    "version": "1.0.0",
    "hostname": "192.168.1.100",
    "port": 2018,
    "remoteioPort": 2019,
    "ssid": "MyWiFi",
    "password": "MyPassword",
    "serialPort": "/dev/ttyUSB0",
    "transports": ["wifi_tcp", "serial"],
    "excludedFiles": [
        ".git/**", "boot.py", "ota.json",
        "host/**", "device/**"
    ],
    "fastOtaFiles": ["main.py"],
    "fullOtaFiles": ["lib/**", "*.py", "*.json"]
}
```

| Key | Description |
|---|---|
| `version` | Version string written to the device after each OTA |
| `hostname` | Device IP for WiFi OTA and RemoteIO connections |
| `port` | OTA server port (default: `2018`) |
| `remoteioPort` | RemoteIO server port (default: `2019`) |
| `ssid` / `password` | WiFi credentials stored on the device |
| `serialPort` | Serial port for bootstrap / serial OTA (auto-detected if empty) |
| `transports` | Active OTA transports on the device (`wifi_tcp`, `serial`) |
| `excludedFiles` | Glob patterns always excluded from OTA uploads |
| `fastOtaFiles` | Files uploaded by `uota fast` |
| `fullOtaFiles` | Files uploaded by `uota full` |

> `boot.py` must stay in `excludedFiles`. It is generated by `bootstrap` and starts the OTA and RemoteIO servers — overwriting it via OTA would break the next boot.

## Boot guard

`boot_guard.py` tracks consecutive unclean boots. On every boot it increments a crash counter in `/ota_boot_state.json`. The OTA server calls `mark_clean()` once running, which resets the counter.

If the device crashes 3 times before `mark_clean()` is called, a warning is printed on the serial console. A future phase will trigger automatic firmware rollback via `esp32.Partition`.

To recover a crash-looping device, re-bootstrap via USB:

```bash
python3 host/uota.py bootstrap
```

## Protected files

The following files are never deleted by the `wipe` command or an OTA manifest diff:

- `ota.py`, `boot_guard.py`, `boot.py`, `remoteio.py`
- `transports/` directory
- `ota.json`, `ota_manifest.json`, `ota_version.json`, `ota_boot_state.json`

## Writing your app

`main.py` is your application entry point. OTA and RemoteIO run in background threads started by `boot.py`.

```python
# main.py
import remoteio

@remoteio.on('status')
def _():
    return {'uptime': time.ticks_ms(), 'led': led.value()}

# Register handlers before the main loop — RemoteIO is already running
while True:
    led.toggle()
    time.sleep(1)
```

If your app is CPU-intensive (tight loops without blocking I/O), add a short startup delay so WiFi has time to connect before your loop takes over:

```python
import time
time.sleep(2)   # let OTA + RemoteIO connect to WiFi
```
